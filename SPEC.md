# makanapo

> ハワイ・ホノルルのカマアイナ割引とハッピーアワー情報を、鮮度高く・検索しやすくまとめる iPhone アプリ。
> ブランド `makana`（贈り物）+ `pō`（夜）＝「宵の贈り物」。

関連: [[CLAUDE]]（Claude Code 用の開発規約）

---

## 1. コンセプト / ポジショニング

- **対象ユーザー**: ホノルル在住のカマアイナ（地元住民）。ハッピーアワーは観光客にも訴求。
- **解く課題**: ハワイは全米屈指の高コスト。割引（カマアイナ/ハッピーアワー）は店ごと・媒体ごとに散在し、しかも古くて信用できない。
- **提供価値**: 「検索できる × 鮮度が高い × 信頼できる × モバイルで使いやすい × 観光ではなく日常特化」。
- **差別化の核**: 各割引に **最終確認日 (`last_verified`)** を表示。放置された既存SEOサイト/リストに対し、鮮度と信頼で勝つ。
- **既存競合との切り分け**:
  - `hawaiihappyhours.com` … ハッピーアワー特化の直接競合 → 「ハッピーアワー単体」では戦わない。
  - 本丸は **カマアイナ割引 + サービス業（サロン/スパ/フィットネス/車/ペット）** の組み合わせ。誰もまとめていない領域。
  - ハッピーアワーは「観光客フック＋日常使いの一機能」として持つ。

## 2. MVP スコープ（意図的に狭く）

- **島**: オアフ島のみ。なんならホノルルのみ。
- **カテゴリ**: 飲食・サービスのみ。
  - 飲食: レストラン → **ハッピーアワー** → カフェ/ベーカリー → ポケ/プレートランチ
  - サービス: 理美容（サロン/床屋）→ スパ/マッサージ → フィットネス/ヨガ → 車（オイル交換/洗車）→ ペットグルーミング
- **最初のフック**: ハッピーアワー（時間依存 × GPS × 高頻度で相性が良い／観光客にも人気）。
- **ローンチ基準**: **「今も有効 ＆ フィールドが埋まる」飲食・サービスの割引が 50 件**。500件の薄いデータより 50〜100件の確認済みデータ。

## 3. アーキテクチャ（実質サーバーレス＝事前ビルド型）

> ライブ/端末側スクレイピングは **しない**。中央で・スケジュールで・1クローラーが収集し、アプリは事前ビルド済みの静的データを読むだけ。

```
[巡回・抽出・正規化・重複排除・期限フラグ・last_verified付与]
   GitHub Actions（毎日 cron）  ※初期はMacのcronでMVPを回しても可
        │ 生成した deals.json を同じリポジトリにコミット
        ▼
   GitHub（データの置き場 / Git によるバージョン管理を兼ねる）
        │ jsDelivr CDN が配信（キャッシュ・高速・無料）
        ▼
   iPhone アプリ（1日1回取得 → 端末にローカルキャッシュ → 検索/フィルタ/地図は端末内で完結）

   ［別系統・書き込み］報告ボタン → Googleフォーム or サーバーレス関数 or Supabase
```

- **持たないもの**: 24時間稼働のアプリケーションサーバー（VPS/EC2のような“箱”）。
- **持つもの**: データの置き場（静的ホスティング）と、報告を受ける小さな受け口（関数/フォーム/BaaS）。常駐プロセスは無い。
- **配信**: 生の `raw.githubusercontent.com` は直叩きせず、**jsDelivr 経由**（`cdn.jsdelivr.net/gh/<user>/<repo>@main/data/deals.json`）。レート制限回避・CDNキャッシュ・無料。GitHub Pages でも可。
- **鮮度の担保**: ライブ取得ではなく **パイプラインの実行頻度**（日次/週次）で担保。
- **Git管理の副産物**: 毎日の差分が履歴に残り、件数の急減（誤爆）を検知・巻き戻しできる。

### なぜライブ/端末側スクレイピングを避けるか
1. 法務: 多数端末が別IPで叩く＝ボット的に見え、契約違反/トレスパスのリスク増。1クローラーがログアウト・レート制限遵守・robots.txt尊重する方が防御的。
2. 品質: 「最終確認日・重複排除・正規化」という中間処理こそが商品。生HTMLを見せるだけでは既存の墓場サイトと同じ。
3. 審査/保守: Apple審査ガイド 5.2（他者コンテンツ再パッケージ）/ 4.2（最小機能）で弾かれやすい。HTML変更でパーサーが壊れても配布済みアプリは即修正できない。

## 4. データ収集戦略

- **基本姿勢**: 「丸ごとスクレイピング」ではなく **一次情報優先**。編集記事・ブログは「どの店が割引中か」を知る **リード** として使い、割引内容・条件は **一次情報（店の公式ページ/公式IG/HVCB/モール公式）で裏取り**し、**説明文は自分の言葉で書く**。
- **収集元（Tier 別）**:
  - **Tier 1（公式・準公式 / 最もクリーン）**: HVCB公式カマアイナ（オアフ）`kamaainaspecialoffers.com/oahu`、アラモアナセンター カマアイナ割引ページ、各店の公式カマアイナ/ハッピーアワーページ。
  - **Tier 2（編集記事 / リードとして使う）**: Honolulu Magazine（カマアイナ/Pau Hana 特集）、HAWAII Magazine、onolicioushawaii.com、nateeatshawaii.com、hawaiihappyhours.com（競合だが網羅性チェックに）。
  - **Tier 3（発見用）**: Yelp / Google（"happy hour Honolulu"）、Instagram（#kamaaina #pauhana）、Reddit r/Hawaii。
  - **Tier 4（オフライン / MVP見送り）**: Star-Advertiser 紙クーポン折込、ワイキキの "Oahu's Best Coupons" 冊子。
- **鮮度の現実**: HVCB等の掲載オファーは期限切れが多く、手動リストも「変わるので要確認」と断っている。→ ソースを鵜呑みにせず、**期限フラグ・未確認ステータスで必ず選別**する。

## 5. データモデル（deals.json の1レコード）

> 確定版スキーマは [[CLAUDE]] と `schema/` に置く想定。以下は人間向けの定義。

| フィールド | 説明 |
|---|---|
| `id` | 一意ID（例: `slug` + ハッシュ） |
| `name` | 店名 |
| `category` / `subcategory` | 飲食/サービス と細分類 |
| `discount` | 割引内容（例: 10% off / カマアイナ価格 / BOGO） |
| `conditions` | 条件（ハワイID提示、店内のみ、酒類除外、ブラックアウト日 等） |
| `redemption` | 利用方法（ID提示 / コード"KAMAAINA" / オンライン） |
| `address` | 住所 |
| `lat` / `lng` | 緯度経度 |
| `neighborhood` | 地区（Kaimuki, Kakaako, Ala Moana 等） |
| `hours` | 営業時間（ハッピーアワーなら時間帯） |
| `source_url` | 情報源URL（一次情報） |
| `last_verified` | **最終確認日（差別化の核・必須表示）** |
| `status` | `active` / `expired` / `unverified` |

## 6. アプリ機能

**v1 に入れる**
- カテゴリ別ブラウズ＆検索
- 地図 + 「近くの割引」（MapKit、追加API費用ゼロ）
- 割引詳細（条件・利用方法・**最終確認日**）
- フィルタ（カテゴリ・地区、可能なら「今やってるハッピーアワー」）
- **「まだ有効？／期限切れ報告」ボタン**（軽量。鮮度をユーザーに維持させる）

**後回し**
- アカウント / 通知 / 期限リマインダー / お気に入り / 加盟店セルフ掲載 / 収益化

**方針**: v1 は **閲覧専用（ログイン不要）**。本人確認データ（州ID画像）は **持たない** → 法務最軽量＆インストール障壁低減。

## 7. 事業モデルの順序（営業は最後）

1. ディレクトリで **ユーザーを集める**（営業なし・公開情報＋クラウド検証）。
2. オアフの日常カテゴリに絞り **密度** を作る。
3. ユーザーが付いてから加盟店がインバウンドで来る → 掲載課金・公式認証バッジ・分析提供で収益化。
- これは「加盟店プラットフォーム型（Groupon型・両面市場）」ではなく **「ディレクトリ/索引型（Googleフライト/RetailMeNot の立ち位置）」**。割引の発行元ではなく索引。

## 8. 法務メモ（要点）

- **著作権**: 事実（価格/割引/住所）は保護対象外。説明文・写真・ロゴ・記事本文は保護対象 → 転載せず自作。一次情報から事実を取り、表現は自分の言葉。
- **スクレイピング**: 公開データのログアウト取得は CFAA 非該当（hiQ/Bright Data）。ただしログイン+規約同意は契約違反リスク。robots.txt 尊重・レート制限。Apple審査の再パッケージ規定に注意。
- **政府データ**: 連邦（NWS等）はパブリックドメイン。州/郡（HVCB等）は利用規約を個別確認。
- **プライバシー（ハワイ）**: 包括法は未成立だが、漏えい通知（HRS 487N）・UDAP（HRS 480-2）あり。位置情報/ブラウザ情報の同意なき販売を禁ずる SB1163 が上院通過。→ **位置情報の扱いに同意設計**、**州ID画像は保持しない**。CA ユーザーが入れば CCPA/CPRA。
- **App Store / FTC**: 位置利用の正当化・プライバシーラベル・ATT。アフィリエイト収益は FTC 開示。安全/健康/金融情報は「公式でない・保証しない」免責。
- **体制**: LLC化で個人責任を遮断、利用規約＋責任制限＋（必要なら）仲裁条項。

## 9. ネーミング決定

- **プロジェクト名 / アプリ名**: `makanapo`（表示は Makanapo / Makana Pō）。
- **意味**: makana（贈り物）+ pō（夜）＝「宵の贈り物」。ハッピーアワー/夜の割引と意味が噛み合う。英語話者・日本語話者ともに発音容易。
- **方式**: 自社ブランド `makana` を冠するハウスブランド方式。`makana` 単独は一般語で混雑だが、`pō` と融合した造語 `makanapo` は識別力が高く所有しやすい。
- **予備クリアランス**: 「makanapo / makana po」に一致するアプリ・企業・商標は検索上見当たらず（実質クリア）。
- **要・本確認（着手と並行で）**:
  - [ ] USPTO `tmsearch.uspto.gov` で "makanapo" / "makana po" を検索（区分: **9 アプリ / 42 SaaS / 35 販促・消費者情報**）
  - [ ] App Store Connect で `Makanapo` の名称予約可否
  - [ ] Instagram 等で @makanapo の空き
